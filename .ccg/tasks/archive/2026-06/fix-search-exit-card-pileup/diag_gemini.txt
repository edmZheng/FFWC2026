## UI Diagnostic Report

### Visual Symptoms
- 用户关闭搜索（`TabBarView` 重建）或折叠赛事日历（`AnimatedSize` 缩小）后，列表卡片异常堆积在 viewport 顶部。
- 只要发生任何微小的手势滑动（触发 `ScrollPosition` 变化），所有卡片瞬间恢复正常的流式布局。

---

### Hypotheses

#### 1. [最有可能] 布局变化（Layout Shift）导致 `_translateY` 状态成为脏缓存
- **Likelihood**: High
- **根因解释**: 
  `StackedEdgeFade` 依赖 `Scrollable.maybeOf(context)?.position.addListener` 来更新 `_translateY`。**但 ScrollListener 只有在 scroll offset (pixels) 变化时才会触发，在 Viewport 尺寸或位置发生变化时（如 `AnimatedSize` 正在收缩）绝对不会触发。**
  当赛事日历收缩或搜索栏关闭时，`TabBarView` 的物理位置在屏幕上整体上移。但在动画期间或重建首帧，卡片依然保留着之前在较低位置时计算出的较大负数 `_translateY`。卡片物理 Y 坐标变小（上移） + 巨大的负数 `_translateY`（再次上移），导致它们全部被拉到了视口的顶部边界。一旦你手动滑动，ScrollListener 终于被触发，重新拿到了稳定后的正确 global 坐标，瞬间将 `_translateY` 修正为 0，布局随之恢复。
- **证据**: “退出赛事日历后也有类似行为” 是决定性证据。退出日历并不销毁 `TabBarView`，只发生 `AnimatedSize` 尺寸变化，说明该 Bug 的核心诱因是**外层布局变化时，卡片未能得到重新计算的机会**。
- **验证方案**: 
  在 `StackedEdgeFade._onScroll` 中加入 `print('translating: $_translateY, pixels: ${position.pixels}, viewport: ${position.viewportDimension}');`。
  观察在点击关闭日历/关闭搜索的瞬间（不手滑），控制台是否没有任何输出，但 UI 已经堆叠。

#### 2. [极大概率] `animateTo(0)` 触发时机过早导致 Viewport 高度为 0
- **Likelihood**: High
- **根因解释**: 
  在关闭搜索时，代码执行了 `setState(() { searchOpen = false; });` 并紧接着调用 `_scrollAllToTop()` (即 `animateTo(0)`)。此时 `TabBarView` 刚刚被塞回 Widget Tree，它的 `RenderViewport` 尚未完成首帧 Layout，其 `viewportDimension` 极有可能为 `0` 或异常值。
  当 `animateTo(0)` 强行在这一帧修改 `pixels` 时，触发了 `StackedEdgeFade._onScroll`。此时计算 `contentBottom` 会得到 0。由于所有卡片的 `idealBottom` 都大于 0，于是所有卡片都计算出 `_translateY = 0 - idealBottom = -idealBottom`，在视觉上刚好完全抵消了它们的原始 Y 坐标，精准地让所有卡片全部重叠在 Y=0（顶部）的位置！
- **证据**: 为什么退出搜索比退出日历“更严重”？因为退出搜索伴随着 `TabBarView` 的整棵树重建 + `animateTo(0)`，不仅丢失了状态，还强行在一个未完全 Layout 的帧里注入了错误的坐标系计算。
- **验证方案**: 
  在 `StackedEdgeFade._onScroll` 的顶部检查并打印：
  `if (position.viewportDimension == 0) print('WARNING: Viewport is 0 during scroll event!');`

#### 3. [中等] `localToGlobal` 参照系穿透了 Scrollable
- **Likelihood**: Medium
- **根因解释**: 
  如果在 `StackedEdgeFade` 中计算 `idealBottom` 采用的是 `renderBox.localToGlobal(Offset.zero)` 而没有传入 `ancestor: scrollableBox`，那么它拿到的是**相对屏幕（物理屏幕）的全局坐标**。
  当顶部带有 `AnimatedSize` 时，卡片在 Scrollable 内部的相对位置没变，但在屏幕上的绝对位置随着日历的展开/收缩在剧烈变化。这导致 `idealBottom` 的计算极度不稳定，容易被提前 Clamp 到底部（然后因为整体布局上移被带到了顶部）。
- **验证方案**: 
  检查 `StackedEdgeFade._onScroll` 中的 `localToGlobal` 传参。如果缺少 `ancestor: Scrollable.maybeOf(context)?.context.findRenderObject()`，则说明参照坐标系错误。

### Probable Root Cause & Lesson Learned

**结论**：这个 Bug 是典型的 **"Flutter Render Metrics 与 State 同步脱节"**。
`StackedEdgeFade` 的视觉位置不仅取决于滚动的 `pixels`，还强依赖于视口的物理高度(`viewportDimension`)以及自身的物理排布(`layoutOffset`)。但原实现只通过 `ScrollPosition.addListener` 监听了 `pixels` 的变化，**完全漏掉了 Layout 变化（AnimatedSize 动画）和首帧 Viewport 为 0 的边界情况**。

一旦发生不带 scroll 事件的 Layout 布局突变，卡片就会带着旧的（或者在 Viewport 为 0 时算出的）脏 `_translateY` 强行渲染，导致偏移严重错乱。这与 `ZSortedListView` 的反向 paint 没有任何关系（反向 paint 是无辜的，它只是恰好让所有被移到顶部的卡片 Z 轴排序发生改变）。

*建议的修复方向：在 `StackedEdgeFade` 中拦截 Viewport 为 0 的计算；或者更彻底地，废弃 `setState` 更新 `_translateY` 的做法，改为实现一个继承自 `SingleChildRenderObjectWidget` 的自定义 RenderBox，在 `paint` 阶段动态读取 ScrollPosition 来进行 `context.paintChild(child, offset + dynamicY)`，这样能彻底免疫任何 Layout Shift 带来的状态不同步问题。*
